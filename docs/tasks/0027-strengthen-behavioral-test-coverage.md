## Goal

Create a dedicated test file for `product-team/src/exec/spawn.ts` covering security-critical functions, and add integration tests for quality-gate lint/test tools that verify real parsing behavior rather than relying exclusively on mocked spawn calls.

---

## Context

Source findings: **D-001** (MEDIUM) and **D-002** (MEDIUM).

**D-001**: `extensions/product-team/src/exec/spawn.ts` has only 32.33% line coverage per the coverage report. Functions `assertSafeCommand()`, `assertPathContained()`, and `parseCommand()` are the same security-critical functions that the quality-gate extension tests extensively in `test/spawn.test.ts`, but the product-team copy has no dedicated test file. If this extension uses its own copy (which it does, pending task 0026), it needs its own tests.

**D-002**: Quality-gate tools mock `safeSpawn` entirely in tests (`lint.tool.test.ts`, `run_tests.tool.test.ts`), so real ESLint/vitest output parsing is never exercised. Parser format drift between tool versions won't be caught.

Note: D-001 may be partially resolved by task 0026 (consolidating spawn to shared contracts). If 0026 lands first, the product-team copy disappears and its test coverage becomes irrelevant. This task should still create/migrate spawn tests to the shared contracts package, and also address D-002 independently.

---

## Scope

### In Scope

- Create `test/exec/spawn.test.ts` in product-team (or migrate to quality-contracts if 0026 is done first) covering: command injection rejection, path traversal rejection, allowlist enforcement, buffer truncation, quote parsing
- Add fixture-based integration tests for `quality-gate/src/tools/lint.ts` using real ESLint JSON output samples
- Add fixture-based integration tests for `quality-gate/src/tools/run_tests.ts` using real vitest JSON output samples
- Fixtures must represent realistic edge cases: empty results, parse errors, mixed stdout/stderr

### Out of Scope

- Running actual ESLint or vitest as part of the test suite (use static fixtures)
- Refactoring the tools themselves

---

## Requirements

1. `product-team/src/exec/spawn.ts` (or its shared-contracts successor) must have ≥ 80% line coverage after this task.
2. At least one integration test per quality-gate tool must use a real fixture file rather than a mocked spawn result.
3. Fixtures must be committed to `test/fixtures/` and be representative of actual tool output.

## Implementation Steps

   If not: add it to `extensions/product-team/test/exec/spawn.test.ts`.

5. Add integration test files that read fixtures and pass them through the tool parsing logic.
6. Run `pnpm test && pnpm lint && pnpm typecheck`.
