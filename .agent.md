# OpenClaw Extensions — Agent Instructions (.agent.md)

## 0) Priority (non-negotiable)
1. Follow this `.agent.md`.
2. Follow the current task file in `docs/tasks/`.
3. Follow `docs/backlog/` for details.
4. If anything conflicts, prefer: correctness > security > simplicity > consistency.

## 0.1) Autonomous Workflow
When the user says **"next task"**:
1. Read `docs/backlog/` to get current task statuses and verify dependencies are DONE.
2. Select the first PENDING task whose dependencies are all met.
3. Create task file: `docs/tasks/NNNN-<title>.md`
4. Create walkthrough file: `docs/walkthroughs/NNNN-<title>.md`
5. Implement the solution.
6. Run quality gates (build, tests).
7. Update walkthrough with results.
8. Commit with conventional format: `feat|fix|test(scope): message`
9. Mark task DONE in backlog.

See [.agent/rules/autonomous-workflow.md](.agent/rules/autonomous-workflow.md) for full details.

## 1) Language
- **English only** in repository content: code, comments, docs, commit messages.

## 2) Repository documentation rules
- Tasks live in: `docs/tasks/NNNN-*.md`
- Walkthroughs live in: `docs/walkthroughs/NNNN-*.md`
- Every task MUST have a walkthrough with the **same file name**.
- If any work is done, the walkthrough must be updated in the same change set/commit.

Naming conventions:
- Standard: `NNNN-...`
- Fast track: `ft-NNNN-...`
- Code review: `cr-NNNN-...`

## 3) Delivery workflow (how work is performed)
For each task:
1. Read the task file and restate the goal in your own words (briefly, in the walkthrough).
2. Implement the smallest correct solution that satisfies acceptance criteria.
3. Keep changes cohesive. Avoid "drive-by refactors".
4. Update the walkthrough with: decisions, trade-offs, commands run, files changed, tests executed, follow-ups.
5. Ensure quality gates pass locally (or document why not).

### 3.1) Task integrity & scope discipline

#### Task Document Preservation
The task document is the immutable specification. Only the Status field and Definition of Done checkboxes may be modified when completing a task. Implementation notes go in the walkthrough, not the task file.

#### Scope Discipline
Only create, modify, or delete files directly required by the task scope. Note out-of-scope ideas in the walkthrough Follow-ups section.

#### Manual Verification
If a task's Testing Plan includes manual verification steps, complete all automated work first, then explicitly ask the user to perform the manual steps before the task is marked DONE.

## 4) Tech stack
- **Runtime**: Node.js 22+
- **Package manager**: pnpm (workspaces)
- **Language**: TypeScript (strict mode, ESM, `.js` extensions in imports)
- **Testing**: Vitest
- **Schemas**: JSON Schema 2020-12
- **Build**: tsc, tsx for CLI execution

```

packages/
  schemas/           # Shared JSON Schemas
skills/

  patterns/          # Architecture patterns skill
docs/
  adr/               # Architecture Decision Records
  patterns/          # Pattern catalog
  tasks/             # Task specifications
  walkthroughs/      # Implementation journals
  backlog/           # Epic/backlog specifications
```

## 6) Extension contract
Extensions live in `extensions/<name>/` and follow the OpenClaw plugin pattern:
- `package.json` with `@openclaw/<name>` naming
- `openclaw.plugin.json` with plugin metadata
- `index.ts` default-exporting `{ id, name, description, register(api) }`
- `register(api)` uses `api.registerTool()`, `api.registerChannel()`, etc.

## 7) Skill contract
Skills live in `skills/<name>/` and contain:
- `SKILL.md` with YAML frontmatter (name, description, metadata)
- Instructions for using the skill's tooling

## 8) Quality gates (Definition of Done)
A task is "DONE" only when:
- Acceptance criteria are met.
- Code builds (`pnpm typecheck`).
- Tests are added/updated and pass (`pnpm test`).
- Lint passes (`pnpm lint`).
- No secrets in repo.
- Walkthrough is updated and accurate.

## 9) Coding standards
- Strict TypeScript, no `any` types
- ESM modules (`"type": "module"`)
- Use `.js` extension in import paths
- Files: `kebab-case.ts`
- Classes/Interfaces: `PascalCase`
- Functions/Variables: `camelCase`
- Constants: `UPPER_SNAKE_CASE`

## 10) Output discipline
- Prefer creating/updating files, not long chat explanations.
- Keep changes minimal but correct.
- Don't invent requirements. If ambiguous, write assumptions in the walkthrough.

## 12) Pull request creation
When the user says **"pr"**:
1. Read `.agent/rules/pr-workflow.md`
2. Verify current branch is not `main`
3. Push to remote if needed
4. Create PR using `gh pr create` with the template from `.github/PULL_REQUEST_TEMPLATE/default.md`
5. Fill ALL sections of the template
6. Report PR URL to user
